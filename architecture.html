How it works: The scouts do their work during dead time — while the user reads the response and types their next message. By the time the next message arrives, intelligent context is already staged. Brain B

An individualized producer chatbot. Manages show preparation, editorial decisions, story analysis, and content production. Completely separate memory and context from Brain A.

Each brain is a fully independent chatbot with its own database, vector index, embedding server, vault, and 6 dedicated agents (3 recall + 3 observer). Three additional shared agents (Memory Gate + 2 Context Scouts) serve both brains. Cross-brain information transfer only happens through the handoff protocol with full provenance tracking. You can add as many brains as you need — each one is a self-contained unit.

Complete Data Flow

From conversation to durable knowledge and back again

1. User speaks to a chatbot
The message arrives. Any pre-staged context from the previous turn's scouts is consumed instantly. The conversation is stored in session state.
2. Memory Gate classifies the message
Immediately and in parallel, the Memory Gate agent classifies the message: none (no memory needed), light (fast embedding search is enough), or deep (trigger full multi-agent recall). Runs in ~3-5 seconds without blocking anything.
3. Passive memory captures explicit saves
If the user says "remember this," the content is embedded (4096-dim vector), stored in the database, indexed in the vector store, and a graph event is enqueued for the brain vault pipeline.
4. Deep recall agents search in parallel (when needed)
When the gate classifies "deep" or the user explicitly asks, three agents fan out simultaneously: Facts does keyword searches, Context reads session history and graph notes, Temporal does date-range queries. Results merge into a ranked payload.
5. Context Scouts pre-stage for next turn
After every response, two Scout agents spin up in background: Topic Trajectory predicts upcoming topics and pre-fetches memories; Relevance Scorer filters and refines current context. Results are staged for the next turn at 0ms latency.
6. Observer agents extract memories continuously
Every 15 minutes during active sessions, three observer agents scan the conversation and extract facts, patterns, and relationship dynamics into durable memory. This happens automatically without the user asking.

Technology Stack

Local-first, privacy-preserving, fully self-hosted

This architecture is model-agnostic. The implementation shown here uses specific models, but the pattern works with any combination of large and small language models. You need two tiers: a primary model (any capable LLM — Claude, GPT, Gemini, Llama, Mistral, etc.) for the main chatbot agents, and a fast model (any cheap, quick model with tool-calling support) for the recall, observer, gate, and scout agents. The embedding layer can use any embedding model. The vector store can be Qdrant, Pinecone, Weaviate, pgvector, or anything that does cosine similarity. The orchestration layer can be any agent framework that supports parallel subagent spawning. The architecture is the idea — swap in whatever models and tools you have.

🧠

Two-Tier Model Strategy

Any capable LLM as the primary chatbot (deep reasoning, large context). Any fast, cheap model for the 13 specialized agents (recall, observer, gate, scouts). Spend tokens where they matter.

🔌

Agent Orchestration

Any framework that can spawn parallel subagents with tool access. The agents need to run concurrently and return structured JSON. Any agent router or framework works.

💾

Vector + Structured Storage

Any vector database for semantic search (Qdrant, Pinecone, pgvector, FAISS) plus any structured store for metadata (SQLite, Postgres, DynamoDB). Per-brain isolation is the key.

🌐

Embedding Layer

Any embedding model that produces consistent vectors. Can be local (MLX, Ollama, llama.cpp) or cloud (OpenAI, Cohere, Voyage). One instance per brain for isolation.

⚙️

Backend API

Any web framework serving memory search, session state, and vault operations. Agents call the API via HTTP tools. Our implementation uses Python FastAPI, but Express, Go, or anything with REST endpoints works.

🖥️

Knowledge Graph Vault

Markdown files with YAML frontmatter, organized in typed folders. Obsidian-compatible. Linked via a graph table for traversal. Any file-based or database-backed note system works.

The architecture is model-agnostic and framework-agnostic — here's the core pattern

If gate says "deep", trigger full recall if gate.classification == "deep": agentic_recall(message, agent_key) # 22s, results staged for next turn def after_response_sent(agent_key, session_id, topic): # Scouts run in background while user reads + types trajectory = spawn_agent("scout-trajectory", tools=["search_memories", "scan_sessions"]) relevance = spawn_agent("scout-relevance", tools=["search_memories", "search_graph"]) results = parallel_run([trajectory, relevance]) staged_buffer.stage(agent_key, results) # ready for next turn

Key Design Principles

  • Agents for comprehension, embeddings for storage. Traditional memory systems embed and retrieve. This system embeds and stores, but uses agents with reasoning to understand what they're retrieving.
  • Multiple perspectives beat single-pass search. A fact search, a context search, and a temporal search each find different things. The merge step combines them into something richer than any single pass.
  • Observation fills the gaps. Users forget to say "remember this." Background observers catch what passive memory misses — patterns, relationships, and facts that matter but weren't flagged.
  • Isolation prevents contamination. Each brain owns its own memory, embeddings, and vault. Cross-pollination only happens through explicit handoff artifacts with provenance tracking.
  • Cheap models for recall, expensive models for reasoning. The recall, observer, gate, and scout agents run on a fast, cheap model. The parent chatbot agents run on a capable model with deep reasoning. You spend tokens where they matter.
  • Graceful degradation. If all three recall agents fail, the system falls back to traditional embedding search. Memory never blocks conversation.
